# Анализ: платформа с вертикалями vs узкие продукты

Дата: 2026-04-21 (вечер, обновлено поздно вечером)
Статус: принято — стратегия Y сохраняется, НО аргументация пересмотрена
после честного возражения основателя.
Связано: `portfolio-strategy.md`, `business-model-review.md`,
`time-budget.md`, `channel-choice.md`.

## Пересмотр 2026-04-21 (поздний вечер) — вопрос основателя

Основатель возразил:
1. «Код пишу не я, а Claude Code — это быстро»
2. «Бренд SecAudit не отталкивает, safety=надёжность»
3. «Дизайн переделаем»

**Это сильные возражения. Пересматриваю честно.**

### Возражение 1: «Код — не bottleneck с Claude Code»

**Частично верно, но с оговорками.**

Что Claude Code ускоряет (×3-5 от ручного):
- Boilerplate (CRUD, handlers, репозитории, миграции)
- Типовые тесты, мок-клиенты
- Рефакторинг при ясной цели
- Интеграции с API по докам

Что Claude Code НЕ ускоряет:
- Архитектурные решения (я даю совет, но решаешь ты)
- Ревизию каждого изменения (читать диффы надо тебе)
- Debug в prod (логи смотришь ты)
- Интервью с ЦА, позиционирование, ICP — это не код
- Маркетинг-контент (статьи/посты) — не код

**Реалистичный пересчёт:** MVP второго продукта без Claude Code = 3-4
мес (это объём при ручном кодинге, ~300-500ч). С Claude Code = 3-4
недели (~70-100ч). Это **огромное** изменение.

**Следствие для multi-vertical:** если код перестаёт быть bottleneck,
то добавление новой вертикали = 1-2 недели кода (раньше было 4-6).

Прошлый аргумент «multi-vertical требует команды» — в части кода **был
слишком абсолютным**. Здесь возражение основателя справедливо.

### Возражение 2: «Бренд SecAudit не токсичен»

**Частично верно, но с нюансом.**

Примеры расширения брендов в РФ:
- **Тинькофф**: банк → брокер → страхование → мобильный оператор → сервисы
  для бизнеса. Сработало потому что (a) бренд про **качество сервиса**,
  не про продукт; (b) расширение в **смежные финансовые** услуги.
- **Яндекс**: поиск → такси → доставка → маркет → музыка. Сработало потому
  что (a) «Яндекс» = generic tech-бренд; (b) каждый продукт — под-бренд
  (Яндекс.Такси).
- **Контур**: ЭДО → бухгалтерия → ИБ-экспертизы. Сработало потому что
  (a) «Контур» generic; (b) 300+ инженеров, 20+ лет.

Против SecAudit:
- «Sec» — не generic, прямо означает Security
- Для **смежных** compliance-вертикалей (HR-compliance документы,
  юридические compliance, 152-ФЗ для медицины) — **бренд работает**, это
  «безопасность документов для соблюдения требований»
- Для **далёких** вертикалей (риэлтор, самозанятый с договором услуг,
  бухгалтер) — **бренд мешает** («при чём тут security?»)

**Ревизия:** бренд SecAudit **не катастрофически токсичен**. Он
**ограничивает расширение в не-compliance вертикали**. Для расширения в
HR/юр/мед — работает. Для самозанятых/риэлторов — мешает, нужен под-бренд
(«SecAudit.ru/samozanyatyy» звучит странно, а «samozanyatyy.secauditai.ru»
— технически работает, но бренд всё равно запутывает ЦА).

Возражение основателя верно в узком смысле (бренд можно расширять в
compliance-соседи), но не в широком (для B2C микро-бизнеса — всё же
слабо).

### Возражение 3: «Дизайн переделаем»

Верно. Дизайн — не bottleneck при Claude Code + дизайн-системе (DESIGN.md
уже есть). 1-2 недели на ре-дизайн в generic-стиле.

### Что ОСТАЁТСЯ как реальный bottleneck

Если код, дизайн и бренд не ограничители, что ограничивает?

**a) Доменная экспертиза (КРИТИЧНО):**
- LLM галлюцинирует на domain-specific вопросах (ФСТЭК №17 отменён, но
  модель подставляет, дата КоАП устарела, «ГОСТ Р 58900-2025» не
  существует — реальные кейсы из промптов SecAudit, см. KNOWLEDGE.md
  историю багов)
- Каждая вертикаль требует **human-in-the-loop** для проверки: ИБ =
  152-ФЗ/187-ФЗ/ФСТЭК. HR = ТК РФ 197-ФЗ. Юристы = ГК РФ. Мед = 323-ФЗ.
- Claude Code может написать промпт, но **не знает вашу предметную
  область**. Ты должен сам проверять каждую правовую отсылку.
- Цена ошибки: предписание регулятора клиенту → иск к автору шаблона →
  гибель репутации

**Время:** на базовую валидацию одной вертикали — 40-80ч изучения
законодательства + 20-40ч настройки промптов через A/B с реальными
пользователями. Это НЕ решается Claude Code.

**b) Маркетинг — ×N на количество вертикалей (КРИТИЧНО):**
- При 24ч/нед бюджета, 12ч/нед на маркетинг
- Один канал качественного контента: **2-4 статьи/мес = 6-10ч/нед**
- 2 канала: 12-20ч/нед → НА ПОТОЛКЕ
- 3 канала: 18-30ч/нед → НЕ ВМЕЩАЕТСЯ
- 5 каналов: 30-50ч/нед → нереально одному человеку

**Это единственный неснимаемый bottleneck при 24ч/нед.** Claude Code не
пишет твои посты на Дзене про самозанятых — он не знает твой тон, твою
ЦА, твой insight.

**c) Customer Success при 50+ юзерах в 3+ вертикалях (КРИТИЧНО):**
- Поддержка, feature requests, отзывы
- Разные вертикали = разные юзеры с разными нуждами
- Интервью, постмортемы оттока — нужны человеко-часы
- Автоматизация через бота помогает на 30-40%, остальное — ручной
  труд

**d) SEO-авторитет (ПРЕОДОЛИМО, но медленно):**
- Один domain с многими вертикальными страницами накапливает авторитет
  быстрее двух доменов
- Но для SEO-поисковых запросов типа «договор оказания услуг самозанятый»
  и «матрица рисков ИБ 152-ФЗ» — алгоритмы Яндекса предпочитают
  точно-релевантные домены
- Это **в пользу 2-х доменов**, не generic-платформы

### Честный итог пересмотра

| Аргумент | Вес в прошлом анализе | Вес после пересмотра |
|---|---|---|
| Код = bottleneck | ⚠️ переоценён | ✅ не bottleneck при Claude Code |
| Бренд «Sec» токсичен | ⚠️ преувеличено | ✅ только для не-compliance |
| Дизайн = barrier | слабый аргумент | ✅ не barrier |
| Экспертиза в вертикалях | недооценён | 🚨 главный bottleneck |
| Маркетинг ×N | правильно оценён | 🚨 главный bottleneck |
| Customer success при росте | не упомянут | 🚨 важный bottleneck |

**Прошлый аргумент «multi-vertical требует команды» был ошибочным в
части кода.** Но реальные ограничители (экспертиза + маркетинг + CS)
по-прежнему **не решаются Claude Code**. Они упираются в человеко-часы
на думание, исследование и разговоры — а не на написание кода.

### Что это меняет в решении?

**Количественно мало. Качественно — добавляет нюанс.**

Для цели 150-200к MRR за 24 мес:
- 2 вертикали (стратегия Y) → реалистично при 24ч/нед
- 3 вертикали (если не писать код вручную) → теоретически возможно, но:
  - Нужно 2-3 мес изучить 3-ю вертикаль (ТК РФ, или ГК РФ, или 323-ФЗ)
  - Нужно 3-й маркетинг-канал развернуть → это +6-10ч/нед от текущих
    24ч → **выход за 28ч/нед лимит** из `anti-burnout.md`
- 5 вертикалей → **нереально** в 24 мес при 24ч/нед даже с Claude Code

**Оптимум остаётся 2 вертикали** — но не потому что «код ограничивает»,
а потому что **маркетинг и экспертиза ограничивают**.

### Правильная формулировка вывода (замена старой)

Старая (ошибочная): *«Multi-vertical требует команды, один человек не
тянет.»*

Новая (точная): **«При 24ч/нед и запрете на найм маркетинг и экспертиза
в нескольких доменах — неснимаемый bottleneck. Claude Code снимает
код-bottleneck, но не остальные. Оптимум = 2 вертикали максимум до M18.»**

## TL;DR

Гипотеза «превратить SecAudit в платформу с разными вертикалями» отклонена
при доступном бюджете 24-26 ч/нед + inbound-only + запрет на личные продажи.

- **Стратегия А (Y)** остаётся оптимальной: SecAudit узко для ИБ +
  AI-шаблонизатор для самозанятых. P(150-200к MRR за 24 мес) = 35-50%.
- **Стратегия Б (платформа с вертикалями)** — 25-45% вероятность, но с
  высоким риском распыления и 60-80 часов «мёртвого» времени на рефакторинг.
- **Стратегия В (гибрид)** — 30-40% и worst-of-both-worlds.

**Важное уточнение в план А:** шаблонизатор проектировать **модульно**
(core AI + domain modules) чтобы на M18+ при success-сигнале можно было
объединить с SecAudit под общий зонтик без переписывания. Не делать
платформу сейчас, но не закрывать дверь архитектурно.

## 1. Анализ потенциальных вертикалей (РФ)

Каждая оценка — реалистичный 18-мес MRR при **inbound-only + один
основатель + 24 ч/нед** на всю деятельность (не только на вертикаль).

### а) ИБ-специалисты (baseline = SecAudit текущий)
- **Рынок:** 300-1k ИП-консультантов + 5k юрфирм с ИБ-практикой + 10-20k
  внутренних (не покупатели из своего кармана)
- **Документы (6):** security_policy, NDA, risk_matrix, IRP, audit_checklist,
  director_letter
- **Частота:** регулярная у консультантов (новые клиенты), разовая у внутренних
- **ARPU:** 2990-14900 (консультантский тариф)
- **Конкуренты:** АльфаДок (3-5k клиентов за 14 лет), SECURITM (enterprise),
  внутренние юротделы
- **Каналы:** Habr (отлично), Telegram @sec_it/@anti_malware_ru/@rusiem, SEO
- **18 мес MRR (реалистично):** 40-120к
- **Потолок:** ~150-250к (узкая ниша)

### б) Юристы / юрфирмы для SMB
- **Рынок:** 170k+ частнопрактикующих юристов + 10k малых юрфирм + 5-7k
  обслуживающих SMB регулярно
- **Документы:** договоры (купли-продажи, услуг, аренды, поставки),
  претензии, иски, заявления, доверенности, учредительные
- **Частота:** высокая (каждый клиент = 5-10 документов)
- **ARPU:** 3000-10000 (привычны платить Гарант/КонсультантПлюс по 50k/год)
- **Конкуренты:** **Гарант, КонсультантПлюс** (справочные базы, слабая
  AI-генерация), FreshDoc, Юрайт, Docsvision
- **Каналы:** профильные Telegram (@lawyermedia, @advokatlibrary), Habr
  (средне), SEO (`шаблон договора купли-продажи`)
- **18 мес MRR:** 80-200к
- **Барьер входа:** высокая цена ошибки (некачественный документ → иск
  от клиента юриста к автору шаблона). Нужен **юр-эксперт на поддержке** —
  один основатель без юр-фона не тянет.

### в) Риэлторы / агентства недвижимости
- **Рынок:** 150-200k риэлторов + 10-15k агентств
- **Документы:** договор купли-продажи, предварительный, аренда, задаток,
  расписка, акт приёма-передачи, агентский договор
- **Частота:** регулярная (каждая сделка = 3-5 документов)
- **ARPU:** 990-2990 (рынок «прижимает» Циан, Авито)
- **Конкуренты:** **ЦИАН** (шаблоны в CRM бесплатно), **Дом.РФ**,
  **Bnovo**, **Alto**
- **Каналы:** Дзен, Telegram риэлторов
- **18 мес MRR:** 30-80к (жёсткая конкуренция с бесплатными шаблонами от
  порталов)
- **Слабое место:** «профессионал-риэлтор» в среднем не покупает отдельный
  софт, если есть бесплатное в Циане

### г) HR / кадровики малых компаний
- **Рынок:** 500k HR в РФ, но **целевой** сегмент «делает документы сам» —
  это HR в компаниях без HRM-системы, 20-40k активных покупателей
- **Документы:** трудовой договор, приказ о приёме/увольнении, должностная
  инструкция, ЛНА, штатное расписание, согласия на ПД сотрудников
- **Частота:** регулярная (раз в месяц новый приём)
- **ARPU:** 2990-7990
- **Конкуренты:** **HRlink**, **Контур.Персонал**, **1С:ЗУП** — сильные,
  но AI-генерация ЛНА почти не представлена
- **Каналы:** Habr (средне), HR-Telegram-каналы, SEO («шаблон трудового
  договора 2026»)
- **18 мес MRR:** 50-150к
- **Барьер:** знание ТК РФ + отраслевая специфика (конкретика в ЛНА
  зависит от вида деятельности)

### д) Бухгалтеры на аутсорсе
- **Рынок:** 50k ИП-бухгалтеров + 10k аутсорс-компаний
- **Документы:** договор услуг, акты, счета, учётная политика, согласия ПД
- **Частота:** очень регулярная
- **ARPU:** 990-2990
- **Конкуренты:** **Контур.Эльба, Моё Дело, СБИС, 1С** — олигополия,
  включая шаблоны
- **18 мес MRR:** 20-60к (рынок перенасыщен)
- **Вердикт:** пропускать. Слишком сильные incumbents.

### е) ИП и самозанятые (выбрано для стратегии Y)
- **Рынок:** 10M+ самозанятых + 4M ИП
- **Документы:** договор услуг, акт, счёт, договор подряда, оферта, НДФЛ,
  согласие ПД
- **Частота:** регулярная (каждый клиент = 2-3 документа)
- **ARPU:** 490-990 (impulse buy)
- **Конкуренты:** **Мой Самозанятый** (шаблоны), **Нимбл**, **Авито Услуги**
  — но AI-генерация слаба, доминирует boilerplate
- **Каналы:** **Дзен (топ-канал для микро-бизнеса)**, Telegram, SEO
  long-tail
- **18 мес MRR:** 60-160к (широкий рынок компенсирует низкий ARPU)
- **Причина выбора:** inbound БЕЗ TikTok работает (Дзен текстовый), широкий
  рынок → трафик при низкой конверсии даёт нужный MRR

### ж) Медучреждения (152-ФЗ, медтайна)
- **Рынок:** 20-30k частных клиник + 10k стоматологий
- **Документы:** ИДС (информированное добровольное согласие), согласие
  на обработку ПД медданных, ЛНА по медтайне
- **Частота:** разово (при открытии/обновлении ФЗ)
- **ARPU:** 4990-14900
- **Конкуренты:** **ДК-Консалтинг**, **1С:Медицина**, специализированные
  медюристы (дорогие консультации)
- **18 мес MRR:** 30-80к
- **Барьер:** отраслевая экспертиза (мединформ + 323-ФЗ + 152-ФЗ
  одновременно). **Без мед-юриста нельзя входить** — риск ошибки высок.

### з) Образовательные учреждения
- **Рынок:** 40k+ школ/садов/ВУЗов (бюджет ограничен) + 200k+
  онлайн-школ/репетиторов
- **Документы:** согласие ПД родителей, устав, положения, договор с
  учащимся, ЛНА
- **ARPU:** 990-4990
- **18 мес MRR:** 20-50к
- **Вердикт:** пропускать. Низкая платежеспособность + сильные incumbents
  (Сферум, ЯКласс).

### и) Транспортные компании
- **Рынок:** 600k перевозчиков (включая ИП)
- **Документы:** путевой лист, ТТН, договор перевозки, акт
- **ARPU:** 990-4990
- **Конкуренты:** **ATI.su, 1С:УАТ, TMS-платформы** — насыщено
- **18 мес MRR:** 20-60к
- **Вердикт:** пропускать. Рынок занят вертикальным SaaS с глубокой
  интеграцией (ATI).

### к) Строительные компании
- **Рынок:** 250k+ компаний, 50k активных
- **Документы:** договор подряда, смета, КС-2, КС-3, ОС-1, журналы
- **ARPU:** 4990-14900
- **Конкуренты:** **Гранд-Смета, Смета.РУ, 1С:Строительство** — доминируют
- **18 мес MRR:** 20-50к
- **Барьер:** смет-экспертиза + интеграция с Гранд-Сметой обязательна,

## 2. Сравнение трёх стратегий

### Стратегия А — текущая Y (два узких продукта)
- SecAudit → ИБ-консультанты (как есть)
- AI-шаблонизатор → самозанятые (новый продукт)
- Два бренда, два домена, два маркетинг-канала

**Технический объём:**
- SecAudit пивот: 2-3 недели (multi-tenant UI + обновление тарифов)
- Шаблонизатор MVP: 3-4 недели (переиспользование 60-70% кода SecAudit)
- Суммарно: 5-7 недель кодинга за M1-M3

**Переиспользование из SecAudit:**
- `internal/ai/*` (провайдеры Anthropic/DeepSeek) — 100%
- `internal/validation/*` (email, password) — 100%
- `internal/handler/auth.go` (рата-лимит, disposable, регистрация) — 90%
- `internal/payment/yookassa/*` — 95% (другие цены)
- `internal/service/document.go` (паттерн async save) — 90% (другие промпты)
- Frontend: `lib/auth.ts`, `lib/passwordValidation.ts`, `lib/api.ts` —
  80-100% (замена endpoint)
- Tailwind дизайн-система (DESIGN.md, токены, компоненты) — 100%
- Инфраструктура: Railway + Vercel + Supabase schema — 100%

**Не переиспользуется:**
- `internal/ai/prompts.go` — 0% (новый домен)
- `internal/legal/*` + `internal/scheduler/legal_update.go` — 0% (только для ИБ)
- `/audit` калькулятор — 0% (ИБ-specific)
- Tech-specific threats, AdditionalParams — 0%

**MRR 24 мес:**
- SecAudit: 40-120к
- Шаблонизатор: 60-160к
- Сумма: 100-280к, таргет 150-200к

**P(150-200к за 24 мес):** 35-50%

**Сильные стороны:**
- Каждый продукт имеет **точный ICP** → сильное позиционирование
- Два независимых канала → диверсификация рисков
- Kill-criteria чистый: один продукт не идёт, второй живёт
- Маркетинг focused (ИБ-аудитория ≠ самозанятые)

**Слабые стороны:**
- Два маркетинг-движка при 12ч/нед на маркетинг = 6ч на продукт → риск
  «недостаточно ни одному»
- Два SEO-домена накапливают авторитет медленно (каждый по отдельности)
- Два бренда = два набора контента, писем, поддержки

### Стратегия Б — SecAudit-платформа с вертикалями
- Один продукт переделан в platform (DocuGen.ru / DocsAI.ru)
- Разделы: `/ib`, `/yurist`, `/samozanyatyi`, `/hr`, `/realtor`
- Пользователь выбирает профессию → свой набор документов
- Общий движок, общий биллинг, общий SEO-домен

**Технический объём:**
- Рефакторинг single→multi-vertical: **3-4 недели**
  - Разделение промптов: `internal/ai/prompts/{ib,yurist,hr,...}.go`
  - Таксономия документов: `domain/vertical.go` с metadata
  - UI: `/onboarding` выбор профессии → персонализация `/generate`
  - Pricing: per-vertical tiers в БД
  - Лендинг-ядро + 5-10 vertical landing pages
- Новая вертикаль поверх платформы: **2-3 недели каждая**
  - Промпты (5-7 новых)
  - Landing page + SEO-страницы
  - Legal RAG наполнение (для юр/мед нужна новая база)
  - Запуск маркетинга в новом канале

**Переиспользование:** 80-90% от SecAudit (больше чем А), но перед этим
3-4 недели «мёртвого» времени на рефакторинг без MRR.

**MRR 24 мес в зависимости от каденции:**
| Вертикалей запущено за 24 мес | Ожидаемый MRR |
|---|---|
| 2 (ИБ + самозанятые, как Y) | 100-280к |
| 3 (+HR) | 130-340к, но нужно 30+ ч/нед на маркетинг |
| 4 (+юристы) | 150-400к, но требует юр-эксперт + 35+ ч/нед |
| 5 (+риэлторы) | 180-450к, но 40+ ч/нед — **нереально одному** |

**P(150-200к за 24 мес):**
- 2-3 вертикали: 30-45%
- 4-5 вертикалей: 20-30% (перегрузка маркетинга при 24 ч/нед)

**Сильные стороны:**
- Общий SEO-домен накапливает авторитет быстрее
- Cross-sell: пользователь HR видит раздел для бухгалтера-аутсорсера и
  пересылает коллеге
- Один биллинг, одна CRM, один саппорт
- Более защищаемый бренд (generic → можно масштабировать дальше)

**Слабые стороны:**
- **3-4 недели рефакторинга = 60-80 часов без движения MRR.** Это -2 мес
  к таймфрейму каждого продукта.
- Generic-бренд слабее для inbound (в SEO/Дзене прямой месседж работает
  лучше: «Шаблон договора самозанятого» vs «Генератор документов»)
- Маркетинг разделён между вертикалями: 12 ч/нед / 3 = 4 ч/вертикаль —
  **недостаточно для traction**
- Каждая вертикаль требует доменной экспертизы → риск галлюцинаций LLM
  выше чем у focused продукта
- Support для 5-10 доменов = пользователь задаёт вопрос про КоАП, а ты —
  ИБ-эксперт, не знаешь. Нужен эксперт в каждой области.

### Стратегия В — гибрид (узкий SecAudit + отдельная платформа)
- SecAudit остаётся узким (для ИБ, в passive после пивота)
- Параллельно **новая** платформа с вертикалями без ИБ

**Технический объём:**
- SecAudit пивот: 2-3 недели (как в А)
- SecAudit в passive с M4: 3-5 ч/нед (контент + поддержка)
- Новая платформа MVP: 6-8 недель с нуля (хуже чем в А, потому что
  заново строим платформенную инфраструктуру)
- Каждая вертикаль в платформе: 2-3 недели

**Переиспользование:** 70-80% из SecAudit (копипаст patterns, но новый
репо → проигрыш по сравнению с А)

**MRR 24 мес:**
- SecAudit (passive): 20-60к
- Платформа с 2-3 вертикалями: 70-150к
- Сумма: 90-210к

**P(150-200к за 24 мес):** 25-40%

**Слабые стороны:**
- **Worst of both worlds:** sunk cost SecAudit (поддерживается), overhead
  платформы, два репо
- Два бренда + несколько вертикалей = максимально расфокусировано
- Saturated: поддержка SecAudit passive + платформа active × 3 = нельзя
  уложиться в 26 ч/нед

## 3. Ответы на ключевые вопросы

### а) Платформа — сила или распыление?

**Примеры успешных multi-vertical SaaS в РФ:**
- **Контур** (Эльба, Персонал, Бухгалтерия, Диадок, Экстерн) — успешен,
  но: 300+ инженеров, отдельные команды на каждую вертикаль, 20+ лет
- **Тинькофф для бизнеса** — 1000+ инженеров, банк+SaaS
- **1С:Предприятие** — платформа с 5000+ разработчиков типовых конфигураций
- **СБИС** — успешен, но строго ЭДО + отчётность (2 вертикали максимум)
- **Битрикс24** — успешен, 200+ инженеров, 15+ лет

**Неуспешные попытки:**
- Множество мелких «all-in-one» SaaS для МСБ вязнут на плато 30-70к MRR
- **Ни одного примера РФ-SaaS, где один человек успешно вёл 5 вертикалей.**

**Вывод:** multi-vertical требует **масштаба команды, не возможен с 1
человеком при 24 ч/нед.** Исключение: если вертикали имеют общий корень
документов (AI-договоры для юристов/риэлторов/ИП — 30-50% документов
общие). Но тогда это «полу-платформа» и она хуже фокуса.

### б) Бренд «SecAudit» на не-ИБ — работает?

**Нет.** «Sec» буквально Security. Юрист/риэлтор/HR не поймёт что это
для него. Расширение токсично.

Для платформы нужен generic бренд. Варианты (брейнсторм):
- **DocuGen.ru** — generic, работает для любой вертикали
- **DocsAI.ru** — подчёркивает AI
- **GenDoc.ru** — коротко
- **ProDoc.ru** — professional documents
- **Formy.ru** — русский, «формы» многозначно

Лучшие кандидаты для inbound-SEO: **DocuGen.ru** (нейтрально), **DocsAI.ru**
(AI как дифференциатор). Проверить домены и индексацию под типовые
long-tail запросы.

Но **в рамках стратегии А это не нужно** — SecAudit узко остаётся,
шаблонизатор получает отдельное имя (например `docset.ru`,
`samozanyatyy.ai`, `shablonai.ru`).

### в) Что мощнее для маркетинга: 2 узких или 1 широкий?

При **inbound-only + 24 ч/нед**:

| Критерий | 2 узких бренда | 1 широкий бренд |
|---|---|---|
| Точность сообщения | высокая | низкая |
| SEO long-tail | средний (2 домена) | высокий (1 домен, много страниц) |
| Контент-time | ×2 | ×1 (но распыление на темы) |
| Авторитет домена | медленно | быстрее |
| Соответствие ICP запросу | точное | generic |
| Kill-criteria простой | да | нет (нельзя killнуть часть платформы) |

**Вердикт:** при 24 ч/нед **2 узких бренда тянутся с трудом, но каждый
точно попадает в ICP.** 1 широкий бренд работает лучше если у владельца
был бы маркетинг-бюджет + SEO-агентство. У нас нет — значит 2 узких выигрывают
**при этих ограничениях**.

### г) Технически: сложность 5-10 вертикалей vs 1

**Узкие места (каждое требует времени поверх базового продукта):**

1. **Промпты** — 5-7 уникальных × 5-10 вертикалей = 25-70 промптов.
   Поддержка: при обновлении законов нужно обновлять все релевантные промпты.
   При 5 вертикалях это +3-5 ч/нед постоянно.

2. **Правовая экспертиза** — ИБ = 152-ФЗ/187-ФЗ/ФСТЭК; HR = ТК РФ 197-ФЗ;
   Юристы = ГК РФ; Медицина = 323-ФЗ; Стройка = ГрК РФ. Без эксперта в
   каждой области LLM галлюцинирует → предписания.

3. **Legal RAG** — для ИБ уже есть. Для HR/юристов/медицины — с нуля.
   Каждая RAG: 20-50 законов, ручная курация, обновления через RSS.
   Работа на 4-6 недель каждая.

4. **UI/UX формы** — разные AdditionalParams для разных вертикалей
   (HR не указывает «SIEM», юрист не выбирает «регуляторы ИБ»). Нужна
   схема «vertical-aware form builder».

5. **Pricing разный** — самозанятые 490, HR 3000-5000, юристы 5000-10000.
   Единый биллинг должен поддерживать per-vertical tiers и ограничения.

6. **SEO** — 5 вертикалей × 20-50 landing pages = 100-250 страниц.
   Автогенерация по шаблону помогает, но нужен контроль качества.

7. **Support** — при 5 доменах пользователи задают вопросы о КоАП, ТК,
   ГК, медтайне. Без эксперта в каждой — отвечает LLM (риск неточности)
   или игнор (churn).

**Сколько сложнее:** для 5 вертикалей поддержка +15-20 ч/нед поверх
базового продуктового времени. **Не влезает в 24-26 ч/нед.**

### д) С какой второй вертикали начать (если платформа)?

По скоринговой модели (рынок × ARPU × низкий барьер × inbound-канал):
1. **Самозанятые** — максимальный рынок, нулевой барьер, Дзен работает
2. **ИБ (baseline)** — уже сделано
3. **HR малых компаний** — средний барьер, high ARPU

**Выбор подтверждает стратегию Y:** самозанятые → второй фокус.
Добавлять HR/юристов/др. **только после M18-M24** когда первые 2
стабильны с 50+ платящими.

## 4. Финальная рекомендация (обновлено после пересмотра)

**Стратегия А (Y) остаётся оптимальной**, но внутри Y есть **три
подварианта**, которые не рассматривались изначально:

### Y(1) — текущий план: два отдельных бренда/домена
- SecAudit (secauditai.ru) остаётся узким для ИБ
- Шаблонизатор — отдельный новый domain (например `docset.ru`,
  `freelance-docs.ru`)
- Два независимых SEO, два бренда, два маркетинг-движка

### Y(2) — ре-брендинг в generic, оба продукта как разделы
- SecAudit ре-брендируется в generic-платформу (например DocsAI.ru)
- Самозанятые — второй раздел на той же платформе
- Общий SEO, один бренд, один маркетинг-движок

Плюсы Y(2):
- SEO накапливается быстрее (один домен vs два)
- Общий биллинг/аккаунты
- Возможность cross-sell между вертикалями

Минусы Y(2):
- **Ре-брендинг SecAudit = 1-2 нед работы** (ре-дизайн, миграция, SEO
  переназначение, обновление Habr/Telegram упоминаний). Это 20-40ч
  «мёртвого» времени без MRR.
- Generic-бренд **слабее для каждой конкретной вертикали** — в Telegram
  @anti_malware_ru сообщение «DocsAI для ИБ-консультантов» менее
  таргетировано, чем «SecAudit — специализированный инструмент для ИБ».
- Накопленный SEO на secauditai.ru теряется (301 редиректы помогают, но
  часть веса всё равно уходит)
- Не даёт сильного преимущества если на 24 мес планируется только 2
  вертикали

### Y(3) — промежуточный: узкие бренды, общий зонтичный бренд (холдинг)
Модель Яндекса: «Яндекс.Такси», «Яндекс.Еда», «Яндекс.Маркет» — каждый
бренд узкий под аудиторию, но все под одним зонтом.

- SecAudit.ru остаётся (узкий бренд для ИБ)
- Шаблонизатор = отдельный узкий бренд, **но тех.инфраструктура общая**
  (один биллинг, один login через SSO, одна админка)
- Архитектура: модульная с первого дня (`core/` + `domains/ib/` +
  `domains/samozanyatyy/`)
- На M18+ если оба работают — можно **объявить зонт** (например
  «SecAudit и [шаблонизатор] теперь входят в группу DocsAI») без
  принудительного ре-брендинга самих продуктов

Плюсы Y(3):
- Сохраняет точность узкого позиционирования для каждой ЦА
- Но даёт возможность объединить бэкофис и маркетинг-активы на M18+
- Не требует ре-брендинга сейчас → нет «мёртвого» времени

Минусы Y(3):
- SEO по-прежнему на двух доменах (авторитет накапливается медленнее
  чем в Y(2))
- Два бренда = два маркетинг-движка (как в Y(1))

### Сравнение

| Критерий | Y(1) текущий | Y(2) generic | Y(3) holding |
|---|---|---|---|
| Ре-брендинг SecAudit | нет | **-20-40ч мёртвых** | нет |
| Точность ЦА-месседжа | ✅ высокая | ⚠️ средняя | ✅ высокая |
| Общий SEO | ❌ | ✅ | ⚠️ частично (позже) |
| Общий биллинг/auth | ❌ (по умолчанию) | ✅ | ✅ (при модульной арх) |
| Риск расфокуса | низкий | низкий | низкий |
| Цена 3-й вертикали M18+ | высокая (новый домен) | низкая | средняя |
| Потеря накопленного SEO | нет | **да, часть** | нет |

### Рекомендуемый выбор: **Y(3) — модульная архитектура, два бренда**

Почему:
- **Сохраняем SecAudit** со всем накопленным авторитетом и доменом
  secauditai.ru
- **Шаблонизатор — новый бренд/домен с собственной ЦА-месседжем** →
  максимум конверсии в нишевом контенте
- **Модульная архитектура** (`core/` + `domains/`) пишется один раз
  в шаблонизаторе → SecAudit-код постепенно мигрирует в эту структуру,
  когда этого требует развитие (не блокирующе)
- **На M18+** если оба работают → можно объединить бэкофис и
  выпустить зонтичный бренд без принудительного ре-брендинга самих
  продуктов

Это **тот же план Y что в `portfolio-strategy.md`**, но с уточнёнными
архитектурными требованиями:
- Шаблонизатор ОБЯЗАТЕЛЬНО пишется с `core/` + `domains/`
- В шаблонизаторе `core/` = auth + billing + AI + polling (тот же что
  можно применить к SecAudit при миграции позже)
- Биллинг/auth можно заранее спроектировать для мульти-доменных
  аккаунтов (один юзер = доступ к нескольким продуктам), даже если
  сейчас использует только один

### Когда Y(3) превращается в Y(2)

Условия для ре-брендинга в generic-платформу:
1. На M18+ оба продукта показали retention ≥50% за 6 мес
2. Есть 3-я вертикаль с подтверждённым product-market fit
3. Общая audience cross-sell сделок ≥10% (юзеры одного = юзеры другого)
4. Есть бюджет времени 40-60ч на миграцию без провала MRR

Если все 4 сойдутся на M18 → `decisions/platform-migration.md` и запуск
ре-брендинга.

### Что НЕ меняется в плане

- AI-шаблонизатор → самозанятые (MVP M2-M3, запуск M3)

### Что ДОБАВЛЯЕТСЯ в план (важно!)

**Архитектура шаблонизатора — проектировать модульно:**

Код делить на:
- `core/` — общий движок: AI-провайдеры, auth, billing, polling, storage
- `domains/ib/` — промпты + формы + landing ИБ (из SecAudit)
- `domains/samozanyatyy/` — промпты + формы + landing самозанятых (новая)

Это значит:
1. SecAudit код **не трогаем структурно сейчас** — он уже работает
2. Новый шаблонизатор пишется в **новом репозитории**, но с чистой
   границей core/domains
3. **На M18+** если оба продукта показали traction (50+ платящих каждый):
   - объединить под одним generic брендом (DocuGen/DocsAI)
   - перенести домены ИБ и самозанятые в общий core
   - добавить 3-ю вертикаль (HR / юристы / другая)
   - это эволюция, не pivot — работает с работающих продуктов

**Ловушка, которую не допустить:** сейчас делать шаблонизатор «как монолит
с самозанятые-only» — через год не сможем добавить HR без переписывания.

**Правильно:** писать с первого дня с `domains/` директорией, даже если в
ней пока один модуль (`samozanyatyy`). Cost: 10-20% дополнительного времени
на архитектуру. Benefit: опция платформы через 18 мес без переписывания.

(из `portfolio-strategy.md`).

  шаблонами-лендингами (если да — SEO-канал забит и надо идти через
  Дзен + long-tail). Это даст указание на M3-M6 маркетинговый план.

### Контрольные точки (без изменений + добавления)

## 5. Антипаттерны — чего НЕ делать

- **Не делать платформу сейчас** (3-4 нед рефакторинга без MRR при
  бюджете 24 ч/нед — это -8% от годового продуктивного времени)
- **Не пытаться запустить 3-ю вертикаль до M18+** (даже если «легко»
  переиспользовать код — маркетинг не тянется)
- **Не переименовывать SecAudit в generic бренд** сейчас (потеря
  накопленного SEO + узнаваемость в ИБ-тусовке)
- **Не копировать код шаблонизатора из SecAudit как монолит-копипаст** —
  писать с `core/domains/` структурой
- **Не добавлять ИБ-вертикаль в шаблонизатор** (оставить SecAudit
  отдельным продуктом до M12 решения)
